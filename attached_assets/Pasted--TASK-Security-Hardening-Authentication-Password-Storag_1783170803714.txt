# PART 1 — Protect ERP APIs

The QA audit confirmed that many ERP endpoints are accessible without authentication.

Apply authentication to every ERP endpoint that should require login.

Requirements:

* Protect all ERP CRUD endpoints.
* Protect reporting endpoints.
* Protect accounting endpoints.
* Protect CRM endpoints.
* Protect procurement endpoints.
* Protect hotel endpoints.
* Protect flight endpoints.
* Protect voucher endpoints.
* Protect administration endpoints.
* Protect AI/OCR endpoints.
* Protect WhatsApp endpoints.
* Protect automation endpoints.

Do NOT protect:

* `/public/*`
* `/portal/*`
* login endpoints
* health endpoints
* any endpoint intentionally designed for public access

Use the existing authentication middleware already implemented in the project.

Do NOT create a second authentication system.

Preserve existing RBAC behaviour.

---

# PART 2 — Password Security

Current passwords are stored in plaintext.

Replace this implementation with bcrypt.

Requirements:

* Store ONLY bcrypt hashes.
* Never store plaintext passwords.
* Never return passwords in any API.
* Never expose password hashes.

Login must compare using bcrypt.

Creating a new user:

plaintext password
↓

bcrypt hash

↓

store hash only

Changing password:

plaintext password
↓

bcrypt hash

↓

replace existing hash

---

# PART 3 — Existing Users Migration

Current database already contains plaintext passwords.

Write a safe migration that:

* Detects plaintext passwords.
* Converts them into bcrypt hashes.
* Preserves the user's existing password.
* Does NOT lock users out.
* Is idempotent (safe to run multiple times).
* Skips accounts already using bcrypt.

---

# PART 4 — Admin Behaviour

Admins must NOT be able to view user passwords.

Replace any "View Password" functionality with:

* Reset Password

Admin workflow:

1. Click Reset Password
2. Enter temporary password
3. Save
4. System hashes password
5. Force password change on next login

If "force change" already exists, reuse it.

If not, implement it with minimal schema changes.
